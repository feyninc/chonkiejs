## Installation Issues

<Accordions>
  <Accordion title="Cannot find module '@chonkiejs/core'">
    Make sure you've installed the package:

    ```bash
    npm install @chonkiejs/core
    ```

    If you're using a monorepo, ensure the package is hoisted or listed in the correct workspace's `package.json`.
  </Accordion>

  <Accordion title="ERR_REQUIRE_ESM or 'import' errors">
    ChonkieJS is an ESM-only package. Make sure your project is configured for ESM:

    - Add `"type": "module"` to your `package.json`, OR
    - Use `.mjs` file extensions, OR
    - Use dynamic `import()` instead of `require()`

    ```json
    {
      "type": "module"
    }
    ```
  </Accordion>

  <Accordion title="TypeScript compilation errors">
    Ensure your `tsconfig.json` uses a modern module resolution:

    ```json
    {
      "compilerOptions": {
        "module": "ESNext",
        "moduleResolution": "bundler",
        "target": "ES2020"
      }
    }
    ```
  </Accordion>
</Accordions>

## Chunking Issues

<Accordions>
  <Accordion title="My chunks are too small">
    Increase `maxChunkSize`. The value represents the maximum token count per chunk.

    ```typescript
    const chunker = new RecursiveChunker({ maxChunkSize: 1024 });
    ```

    Common values:
    - **256** — tight context windows, Q&A bots
    - **512** — balanced default for most RAG pipelines
    - **1024+** — longer documents, summarization tasks
  </Accordion>

  <Accordion title="My chunks are too large">
    If chunks exceed your expected size:

    1. `maxChunkSize` is in **tokens**, not characters
    2. Some chunkers (SentenceChunker, RecursiveChunker) won't split mid-sentence
    3. For strict limits, use `TokenChunker` which guarantees the size constraint
  </Accordion>

  <Accordion title="SemanticChunker requires an embedding function">
    SemanticChunker needs you to provide an `embeddingFn` that maps text to vectors:

    ```typescript
    import { SemanticChunker } from '@chonkiejs/core';

    const chunker = new SemanticChunker({
      maxChunkSize: 512,
      similarityThreshold: 0.5,
      embeddingFn: async (texts: string[]) => {
        // Call your embedding API here
        // Must return number[][] (one vector per input text)
        const response = await fetch('your-embedding-api', {
          method: 'POST',
          body: JSON.stringify({ inputs: texts })
        });
        return response.json();
      }
    });
    ```
  </Accordion>

  <Accordion title="CodeChunker produces empty chunks or errors">
    Common causes:

    1. **Wrong language** — make sure the `language` parameter matches your file (`"typescript"`, `"python"`, etc.)
    2. **Invalid syntax** — the AST parser can't handle files with syntax errors
    3. **Very small files** — if the file is smaller than `maxChunkSize`, you get a single chunk
  </Accordion>

  <Accordion title="FastChunker cuts in the middle of words">
    Expected behavior. FastChunker splits on byte boundaries for maximum throughput. Use `RecursiveChunker` or `SentenceChunker` if you need clean word/sentence boundaries.
  </Accordion>
</Accordions>

## Integration Issues

<Accordions>
  <Accordion title="Embedding dimension mismatch with vector DB">
    Your embedding function's output dimension must match your vector database's configured dimension.

    Common dimensions:
    - OpenAI `text-embedding-3-small`: 1536
    - Cohere `embed-english-v3`: 1024
    - Local models vary — check your model's docs

    Verify your handshake was created with the correct dimension:

    ```typescript
    const handshake = new QdrantHandshake({
      collectionName: 'my-docs',
      dimension: 1536, // Must match your embedding output
    });
    ```
  </Accordion>

  <Accordion title="Handshake connection errors">
    Verify:

    1. Your vector DB is running and accessible
    2. API keys are correct
    3. The collection/index exists (check your DB's dashboard)
    4. Network allows the connection (CORS for browser, firewall for server)
  </Accordion>

  <Accordion title="Memory issues with large documents">
    For very large documents:

    1. **Pre-split** — divide into sections before chunking
    2. **Process sequentially** — avoid loading all chunks into memory at once
    3. **Stream to vector DB** — write chunks as they're produced

    ```typescript
    // Process in batches
    const sections = splitLargeText(document, 100_000);
    for (const section of sections) {
      const chunks = await chunker.chunk(section);
      await handshake.write(chunks);
    }
    ```
  </Accordion>
</Accordions>

## Tokenizer Issues

<Accordions>
  <Accordion title="Custom tokenizer not counting tokens correctly">
    If you're providing a custom tokenizer, make sure it returns accurate token counts. The chunkers rely on these counts to respect `maxChunkSize`:

    ```typescript
    import { TokenChunker } from '@chonkiejs/core';

    const chunker = new TokenChunker({
      maxChunkSize: 512,
      tokenizer: {
        encode: (text) => /* must return token array */,
        decode: (tokens) => /* must return string */,
      }
    });
    ```

    Test your tokenizer independently to verify counts match expectations.
  </Accordion>
</Accordions>
